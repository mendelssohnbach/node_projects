# リファクタリング学習メモ

リファクタリング・モジュール分割などを検討した際の判断視点を記録していく。

## 2026-07-13: `promptGet.js` のモジュール分割視点

### 対象
`promptGet.js`（バリデーション・`Person`データモデル・CSV永続化・CLI入出力・アプリケーションループが1ファイルに同居している状態）

### 判断視点

1. **責務（単一責任）**
   異なる関心事（バリデーション／データモデル／CSV永続化／CLI入出力／アプリケーションフロー）は変更理由が異なる。バリデーションルールの変更とCSV出力形式の変更は無関係なので、責務単位で切るのが素直。

2. **I/O依存 vs 純粋ロジック**
   `emailRegex`/`numberRegex` によるバリデーションは入力さえあればテストできる純粋関数。一方 `prompt`（標準入力）や `csv-writer`（ファイル書き込み）は副作用を伴うI/O層。分離すると純粋ロジック部分が単体テストしやすくなる。

3. **依存の向き**
   ドメイン（`Person`）が特定のI/Oライブラリ（`csv-writer`）を直接知っている状態は結合が強い。永続化ロジックを外に出せば、`Person`はデータ構造に専念でき、保存先の差し替えなどの変更にも強くなる。

### 結論（採用した分割案）
まずは3分割から着手する。
- バリデーション
- `Person`（＋CSV保存）
- メインループ

## 2026-07-13: バリデーション部分を切り出す作業をやってみた（初心者向け記録）

### 今回やったこと
`promptGet.js` という1つの大きなファイルの中に混ざっていた「入力が正しいかどうかをチェックする部分」を、`validators.js` という新しいファイルに引っ越しさせました。

### たとえ話で説明すると
`promptGet.js` は、いろんな道具（レジ、はかり、包丁など）が一つの引き出しにごちゃまぜに入っている状態でした。今回は、その中から「はかり」にあたる部分――つまり「入力された電話番号やメールアドレスが正しい形かどうかを測る道具」――だけを取り出して、専用の引き出し（`validators.js`）に移した、というイメージです。

はかりだけ別の引き出しにしておくと、
- はかりが正しく動くかどうかを、他の道具を出さずに単独で確認できる
- はかりの調子が悪いときに、どこを直せばいいか一目でわかる

というメリットがあります。

### 具体的に何を変えたか
- 新しいファイル `validators.js` を作った
  - 電話番号が数字だけかどうかを確認する `isValidNumber`（電話番号専用の「チェック係」）
  - メールアドレスが「◯◯@◯◯.◯◯」のような形になっているかを確認する `isValidEmail`（メール専用の「チェック係」）
  - この2つのチェック係を、外のファイルからも呼び出せる状態にして置いた
- `promptGet.js` の中に直接書かれていたチェックの仕組みは削除し、代わりに `validators.js` の2つのチェック係を呼び出す形に変更した

### 図解: ファイルの中身がどう変わったか

作業前（1つの引き出しに全部入っていた状態）
```
promptGet.js
├── メールの形式チェック
├── 電話番号の形式チェック
├── Personの入れ物（データ保管・CSV保存）
├── 入力受付・チェック呼び出し
└── アプリのメインループ
```

作業後（チェック係だけ別の引き出しに引っ越した状態）
```
validators.js
├── isValidNumber()  ← 電話番号のチェック係
└── isValidEmail()   ← メールのチェック係

promptGet.js
├── Personの入れ物（データ保管・CSV保存）
├── 入力受付（validators.js のチェック係を呼び出すだけ）
└── アプリのメインループ
```

### ついでに見つかった不具合の修正
作業中、`promptGet.js` の中で、CSVファイルがすでにあるかを確認する行（`fileExistsAndNotEmpty`）が、今回の作業とは別の理由でうっかり消えてしまっていることに気づきました。今回の作業とは関係のない箇所でしたが、そのままだとアプリが動かなくなるため、ユーザーに確認したうえで元の内容に戻しました。

### 動作確認したこと
実際にアプリを動かして、次の2パターンを確認しました。
1. わざと間違った電話番号（例: `abc`）を入力する → 「Error: Phone number should contain only digits.」というエラーが表示され、最初の入力からやり直しになる
2. 正しいデータを入力する → 「◯◯ Saved!」と表示され、`contacts.csv` に正しく保存される

どちらも、引っ越し作業をする前と同じ動きになっていることを確認できました（確認用に作ったファイルは後片付け済み）。

### 次にやること
チェック係（`isValidNumber` / `isValidEmail`）が単独で正しく動くことを、`node:test` という仕組みを使って自動で確認できるようにする（`validators.test.js` を同じ階層に作る予定）。

## 2026-07-13: チェック係の自動テストを作った

### 今回やったこと
`validators.js` に引っ越しさせた2つのチェック係（`isValidNumber` / `isValidEmail`）が、本当に正しく動いているかを自動で確認する仕組み（自動テスト）を作りました。

### たとえ話で説明すると
自動テストは「チェック係の採用試験」のようなものです。「123」という数字を渡したら合格（true）と答えるか、「abc」という文字を渡したら不合格（false）と答えるか――あらかじめ用意した何十個もの問題を毎回自動で出題し、正しく答えられるかを確認します。人が毎回手で試す代わりに、コンピュータに試験官をやらせるイメージです。

### 具体的に何をやったか
- `validators.test.js` というファイルを、`validators.js` と同じ場所に作成した
- Node.js に標準で入っている `node:test` という試験官の仕組みを使用した
- 電話番号チェックについて9パターン、メールチェックについて9パターン（合計18パターン）の問題を用意し、それぞれ正しい答えが返ってくるかを確認した
  - 例: 数字だけ→合格、文字が混ざっている→不合格、空文字→不合格、マイナス符号→不合格 など
  - おまけとして、現在のメールチェックの仕組みでは `"a b@c.com"` のような一見おかしな文字列も合格になってしまう、という現状の仕様もそのまま記録した
- `package.json` の `test` コマンドを、この試験官が動くように設定変更した（`npm test` で実行できるようにした）

### 結果
`npm test` を実行し、18問すべて合格することを確認しました。

## 2026-07-13: Personクラス（＋CSV保存）を引っ越しさせた

### 今回やったこと
`promptGet.js` に残っていた「連絡先データの入れ物（`Person`クラス）」と「CSVファイルへの保存の仕組み」を、`person.js` という新しいファイルにまとめて引っ越しさせました。

### たとえ話で説明すると
`Person`クラスは「連絡先カードを1枚作る係」、CSV保存の仕組みは「そのカードをファイルに綴じ込むファイリング係」です。この2つは常にセットで動く（カードを作ったら必ずファイルに綴じる）ので、前回のチェック係とは違い、あえて分けずに同じ引き出し（`person.js`）にまとめて入れました。

一方、`promptGet.js` の方は「お客さんに質問して回答を受け取る受付係」と「受付からファイリング係を呼び出す進行役」だけが残る形になり、見通しがすっきりしました。

### 具体的に何を変えたか
- 新しいファイル `person.js` を作った
  - CSV保存先のパス、ファイルがすでにあるかどうかの確認、`csvWriter`（ファイリング係の道具）
  - `Person`クラス（連絡先カードを作る・そのカードを保存する）
  - これらをまとめて、外から使えるように `export` した
- `promptGet.js` からは、上記のコードをすべて削除し、代わりに `person.js` から `Person` を読み込む1行に置き換えた
  - あわせて、不要になった `csv-writer` と `node:fs` の読み込みも削除した

### 図解: ファイルの中身がどう変わったか

作業前
```
validators.js
├── isValidNumber()
└── isValidEmail()

promptGet.js
├── Personの入れ物（データ保管・CSV保存）
├── 入力受付（validators.js のチェック係を呼び出すだけ）
└── アプリのメインループ
```

作業後
```
validators.js
├── isValidNumber()
└── isValidEmail()

person.js
└── Personクラス（連絡先カード作成 ＋ CSVへの保存）

promptGet.js
├── 入力受付（validators.js のチェック係を呼び出す）
├── person.js から Person を呼び出して保存を依頼
└── アプリのメインループ
```

### 動作確認したこと
実際にアプリを動かし、名前・電話番号・メールアドレスを入力 → 「◯◯ Saved!」と表示され、`contacts.csv` に正しい内容で保存されることを確認しました。また、既存の自動テスト（18問）もすべて合格することを確認し、今回の引っ越しが他の部分に影響していないことを確かめました。

残る分割は「メインループ」のみ。3分割案（バリデーション／Person＋CSV保存／メインループ）のうち、2つが完了しました。

## 2026-07-13: メインループまわりの整理

### 今回やったこと
`index.js` は変わらず `promptGet.js` の `promptGet()` を呼び出す形（入口）のままにする、という前提のもと、新しいファイルは作らず `promptGet.js` の中身の見た目だけを整理しました。

### たとえ話で説明すると
バリデーション係・ファイリング係が別の部屋に引っ越したあと、残った部屋（`promptGet.js`）には「受付」と「進行役」だけが残りました。ただ、その部屋の入り口に、もう別の部屋に移動した「ファイリングの見本（CSVの出力例）」の貼り紙がそのまま残っていたので、正しい部屋（`person.js`）に貼り直しました。

### 具体的に何を変えたか
- CSVの出力例を示すコメント（`NAME,NUMBER,EMAIL,CREATED_AT...`）を `promptGet.js` から `person.js` のCSV書き出し設定の近くに移動した
  - 理由: CSVを実際に書き出しているのは `person.js` なので、書式の説明もそちらにある方が自然
- `promptGet()` 関数自体に「連絡先の入力受付から保存、継続確認までのメインループを開始する」という説明（JSDoc）を追加した
  - 理由: 内部の `promptWithValidation` / `startApp` にはすでに説明があったが、外側の `promptGet()` にはなく、説明の付き方が不揃いだった

### 図解: 最終的なファイル構成
```
validators.js
├── isValidNumber()
└── isValidEmail()

person.js
├── CSV出力の書式（出力例つき）
└── Personクラス（連絡先カード作成 ＋ CSVへの保存）

promptGet.js  ← index.js から呼ばれる入口はここのまま
├── 入力受付＋再入力ループ（promptWithValidation）
└── アプリの進行制御（startApp）

index.js
└── promptGet() を呼び出すだけ（変更なし）
```

### 動作確認したこと
アプリを実際に動かし、保存成功（`dave Saved!`）とCSVへの正しい書き込みを確認しました。既存の自動テスト（18問）もすべて合格し、影響がないことを確認しました。
